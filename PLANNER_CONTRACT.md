# Контракт: сервер ↔ ИИ-планировщик (раскладка по принтерам)

---

## Роли и правила

**Сервер** определяет состав и формат передаваемых данных: и тех, что отправляет ИИ-агенту, и формата ответа ИИ. Схема данных и правила задаются сервером.

**ИИ-агент** обязан строго следовать схеме данных и правилам, установленным сервером. Задача ИИ — выполнить **планирование печати**: распределить файлы печати по принтерам с учётом указанных требований (в том числе дополнительных из `policy`) и вернуть список файлов печати с указанием:
- принтера (обязательно выбрать из доступных для этого файла печати);
- даты и времени начала печати.

Данные, которые ИИ-агент передаёт серверу, должны быть **упакованы в схему данных**, заданную сервером. ИИ-агент не передаёт никаких других данных, кроме запрошенных сервером по схеме; при необходимости что-то пояснить или добавить — только в описательной части (например, поле `reasoning`).

---

## Текущая передача данных (заглушка)

Сервер отправляет ИИ один JSON-объект в теле сообщения пользователя:

- `request_id` — UUID запроса
- `schema_version` — версия (например `"1.0.0"`)
- `tasks` — массив задач; в каждом элементе обязательно:
  - `task_id` — целое число, индекс задачи от **0** до **N−1** (порядковый номер в массиве `tasks`; N = длина массива). Эти же значения ИИ должен вернуть в ответе.
  - остальные поля задачи задаются отдельно
- `printers` — массив принтеров `{ "id", "name" }`
- `materials` — массив материалов `{ "id", "name", "color", "plastic_type" }`
- `print_jobs` — массив файлов печати; каждый элемент:
  - `id` — идентификатор файла печати
  - `name` — наименование файла печати
  - `available_printer_names` — список наименований доступных принтеров для этого файла
  - `execution_time` — время выполнения
  - `parts_count` — количество деталей (колонка «Детали»)
- `policy` — необязательно, текст указаний пользователя

**Связь задачи и файла печати:** через детали. В задаче описаны детали (какие детали и в каком количестве), в файлах печати — тоже (состав по деталям). По совпадению деталей задача соотносится с файлом печати.

---

## Что получаем от ИИ-агента

Сервер ожидает от ИИ один JSON-объект в ответ:

- `request_id` — эхо `request_id` из запроса (для сопоставления)
- `response_version` — версия формата ответа (например `"1.0.0"`)
- `reasoning` — краткое объяснение решений (строка, до 2000 символов)
- `assignments` — массив назначений; каждый элемент:
  - `task_id` — **то же значение**, что в запросе: целое число 0..N−1 (индекс задачи из массива `tasks`). ИИ обязан возвращать именно эти идентификаторы без изменений.
  - `printer_id` — выбранный принтер (id из переданного списка `printers`)
  - `start_at` — **обязательно**: дата и время начала задачи. Формат — как в API сервера: строка в **ISO 8601** с указанием часового пояса (например `2026-03-03T10:00:00+03:00`). Ограничение: значение должно попадать в **выбранную неделю** плана (от `week_start` 00:00:00 включительно до `week_end` 00:00:00 исключительно; границы недели передаются в запросе в элементах `tasks` или в общих полях).
  - при необходимости: порядок, коды обоснования
- `unassigned` — необязательно: массив задач, которым не назначен принтер; в каждом элементе `task_id` (те же 0..N−1) и код причины. Список кодов причин задаётся отдельно.
- `confidence` — необязательно: уверенность 0..1
- `warnings` — необязательно: массив строк-предупреждений. Сюда же: если задач слишком много и не влезают в выбранную неделю — задачи, перенесённые на следующую неделю, перечисляются в `warnings`.

**Правила планирования (ИИ обязан соблюдать):**
- Использовать **только разрешённые принтеры**: для каждой задачи (файла печати) принтер обязан быть из списка доступных для этой задачи; другие принтеры использовать нельзя.
- `start_at` должен попадать в **выбранную неделю** (от `week_start` 00:00:00 включительно до `week_end` 00:00:00 исключительно).
- Если задач столько, что в выбранную неделю они не влезают — продолжать планирование на **следующую неделю** и выводить список задач, перенесённых на следующую неделю, в `warnings`.

Детальная схема полей и правила валидации на сервере задаются отдельно.
