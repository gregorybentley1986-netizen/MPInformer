## Логика работы «Очереди поставок» (MPI ↔ Ozon)

Этот документ — «истина» для поведения страницы `Очередь поставок` и связанных API.

### Сущности

- **MPI** хранит поставки в таблице `ozon_supplies` (`OzonSupply`).
- **Ozon ЛК / Seller API** — источник правды для:
  - фактического состава заявки (какие SKU и в каком количестве реально допустимы к приёмке),
  - статуса заявки,
  - состава грузомест (cargoes) и их содержимого.

### Общий сценарий (как должно работать)

#### 1) Создание поставки

- Пользователь в UI вводит **SKU + количество**, выбирает **кластер** и **дату**.
- UI вызывает `POST /api/supplies/draft`.
  - MPI создаёт **черновик** в Ozon.
  - MPI запрашивает **свободные таймслоты** для выбранных кластеров/дат.
  - UI показывает слоты пользователю.

#### 2) Подтверждение слота → создание заявки

- Пользователь выбирает таймслот.
- UI вызывает `POST /api/supplies/confirm`.
  - MPI подтверждает черновик → создаётся **заявка на поставку**.
  - MPI создаёт строку `OzonSupply` в БД (с заявленным составом).
  - Через несколько секунд запускается **фоновая сверка** состава с ЛК Ozon.

#### 3) Фоновая сверка состава (важно)

Цель: учесть, что Ozon может принять не все товары.

- Фоновое обновление должно подтянуть из ЛК:
  - **актуальный состав** по bundle_id (через `/v3/supply-order/get` → bundle_id → `/v1/supply-order/bundle`);
  - **статус заявки**;
  - **грузоместа** (`/v1/cargoes/get`) и их содержимое (при необходимости — дополнительно через bundle_id каждого грузоместа).

Результат:

- Если актуальный состав из ЛК **НЕ совпадает** с заявленным в MPI:
  - показываем предупреждение,
  - показываем кнопку **«Скорректировать состав»**.

#### 4) «Скорректировать состав»

- Нажатие вызывает `POST /api/supplies/{id}/content-update`.
- MPI:
  - ещё раз берёт актуальный состав из ЛК (bundle),
  - отправляет полный состав в Ozon (`/v1/supply-order/content/update`),
  - ждёт статус операции,
  - при `SUCCESS` обновляет `composition` в MPI и убирает предупреждение.

После этого состав в MPI должен стать **точно таким же, как в ЛК Ozon**.

#### 5) Грузоместа

- Если состав корректный → пользователь нажимает **«Создать грузоместо»** / **«Редактировать грузоместа»**.
- При отправке грузомест:
  - MPI сохраняет введённую пользователем структуру грузомест,
  - отправляет в Ozon (`/v1/cargoes/create`),
  - затем в фоне опрашивает статус заявки до **READY_TO_SUPPLY** (готово к отгрузке).

Проверки:

- MPI должен проверять, что **все товары распределены по грузоместам**:
  - если не распределены — выводим предупреждение: **SKU + кол-во** остатка.
- MPI должен отображать, что статус заявки в ЛК — **«Готово к отгрузке»** (READY_TO_SUPPLY).

### Обновление строки в UI (AJAX)

Автоматический опрос **всех** строк при загрузке страницы отключён (лимиты Ozon «per second»). Обновление статуса/предупреждений по одной строке — через `POST /api/supplies/{id}/refresh` при действии с этой поставкой (например, после запроса этикеток грузомест) или после полной перезагрузки страницы (F5 / сохранение грузомест / синхронизация состава).

### Ссылки на код

- **UI**: `templates/site/supply_queue.html`
- **API**:
  - `POST /api/supplies/draft` — создание черновика
  - `POST /api/supplies/confirm` — создание заявки
  - `POST /api/supplies/{id}/refresh` — синхронизация с ЛК Ozon
  - `POST /api/supplies/{id}/content-update` — «Скорректировать состав»
