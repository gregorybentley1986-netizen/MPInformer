# Как смотреть логи MPInformer

Логи пишутся в **stdout** (стандартный вывод). Куда они попадают — зависит от того, как вы запускаете приложение.

---

## 1. Запуск в том же терминале (передний план)

Если запускаете так:
```bash
cd /path/to/MPInformer
python main.py
```
или
```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

**Логи идут прямо в этот терминал.** Ничего дополнительно делать не нужно — все сообщения (в т.ч. по Ozon/WB) будут в этом окне.

Чтобы после нажатия «Обновить Ozon» увидеть наши отладочные строки, просто смотрите в тот же терминал, где запущено приложение.

---

## 2. Запуск в фоне (nohup, &)

Если запускаете так:
```bash
nohup python main.py &
```
логи по умолчанию попадают в файл **`nohup.out`** в текущей директории.

Смотреть логи в реальном времени:
```bash
cd /path/to/MPInformer
tail -f nohup.out
```

Или последние 100 строк:
```bash
tail -n 100 nohup.out
```

Поиск по логам (например, только Ozon):
```bash
grep -i ozon nohup.out
tail -f nohup.out | grep -i ozon
```

---

## 3. Запуск через systemd (сервис)

Если приложение запускается как сервис (например, `systemctl start mpinformer`), логи забирает **journald**.

Смотреть логи в реальном времени:
```bash
journalctl -u mpinformer -f
```
(имя сервиса может быть другим — то, как назвали `.service` файл)

Последние 200 строк:
```bash
journalctl -u mpinformer -n 200
```

Только за сегодня:
```bash
journalctl -u mpinformer --since today
```

Поиск по тексту (Ozon, warehouse, stocks):
```bash
journalctl -u mpinformer -f | grep -i ozon
```

---

## 4. Запуск в screen или tmux

- **screen:**  
  Запустить: `screen -S mpi`, затем `python main.py`.  
  Отключиться: `Ctrl+A`, затем `D`.  
  Вернуться и смотреть логи: `screen -r mpi`.

- **tmux:**  
  Запустить: `tmux new -s mpi`, затем `python main.py`.  
  Отключиться: `Ctrl+B`, затем `D`.  
  Вернуться: `tmux attach -t mpi`.

Логи будут в том же окне session при следующем подключении.

---

## Что искать в логах после «Обновить Ozon»

После нажатия «Обновить Ozon» в логах должны появиться строки вида:

- `Ozon warehouse/list: ...` — ответ API списка складов
- `Ozon stocks v4: ключи ответа=...` — структура ответа остатков
- `Ozon stocks v4: ключи первого item=...` — поля каждого элемента (есть ли warehouse_id, stocks и т.д.)
- `Ozon stocks by_cluster: всего складов/кластеров=...` — сколько кластеров и артикулов разобрано
- `Ozon stocks by_cluster: итого кластеров=...` — финальное количество кластеров и строк

Если видите `Ozon stocks v4: первая порция пуста (items=0)` — API вернул пустой список остатков.  
Если видите `Ошибка при получении остатков Ozon по кластерам` — ниже будет текст исключения (таймаут, 4xx/5xx, разбор JSON и т.д.).

## Логи ИИ при переносе плана в задачи (ИИ)

При нажатии «Перенести план в задачи (ИИ)» сервер отправляет запрос к LLM (Ollama и т.п.) и пишет в лог:

- **LLM printer assignment: sending request** — отправлен запрос, число задач и длина промпта.
- **LLM printer assignment: raw response (first 600 chars)** — первые 600 символов ответа модели (её «рассуждения» и JSON).
- **LLM printer assignment: success** — модель вернула валидный JSON, расстановка по ИИ применена.
- **LLM printer assignment: validation failed** — модель ответила, но формат не подошёл (нет `task_index`/`printer_id` или принтер не из списка). Используется алгоритм.
- **LLM printer assignment failed (fallback to algorithm)** — ошибка (сеть, таймаут, неверный JSON). Используется алгоритм.
- **Transfer-to-queue: use_ai=True but LLM returned None** — запрос с ИИ был, но в итоге расстановка по алгоритму.

**Где смотреть:** там же, где общие логи приложения (терминал, `journalctl -u mpinformer -f`, `nohup.out` и т.д.). На **сервере**, где крутится MPInformer, а не на ПК с Ollama — логи пишет именно сервер при вызове ИИ.

---

## Уровень логирования

По умолчанию используется уровень **INFO** (наши отладочные сообщения как раз INFO).

Чтобы включить более подробные логи (DEBUG), в админке: **Настройки** → **Уровень логирования** → **DEBUG**, сохранить и при необходимости перезапустить приложение. После отладки лучше вернуть **INFO**.
